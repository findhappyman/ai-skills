# 文章优化技能 (Article Optimizer)
 
## 概述
 
将用户提供的中文文章优化为通俗易懂、逻辑通顺的版本，并分别生成适合微信公众号、X（推特）、小红书三个平台发布的内容。同时为每个平台版本提供创意配图的 AI 生图 prompt。
 
最终输出为一个 Markdown 文件供用户下载。
 
---
 
## 工作流程
 
### 第一步：分析原文
 
收到用户文章后，先进行分析（不要输出分析过程，直接用于指导优化）：
 
1. **主题提炼**：文章的核心观点是什么？
2. **受众判断**：文章面向什么样的读者？
3. **风格识别**：原文是正式/学术/口语/叙事/议论？
4. **问题诊断**：
   - 是否有逻辑跳跃或断裂？
   - 是否有晦涩难懂的表达？
   - 是否有冗余或重复？
   - 结构是否清晰？
### 第二步：优化原文
 
基于分析结果优化文章，遵循以下原则：
 
- **保留原意**：这是最高优先级。不改变作者的观点和立场
- **通俗易懂**：用更简单的词汇替换晦涩术语（必要时保留术语但加解释）
- **逻辑通顺**：确保段落之间有清晰的逻辑衔接
- **精简冗余**：删除重复表达，但不过度删减导致信息丢失
- **语感自然**：读起来像人写的，不像机器翻译或 AI 生成的
- **必要时调整结构**：如果原文的段落顺序不利于阅读体验或论点展开，可以重新编排章节顺序（例如将核心观点前置、将论据或案例后移），但调整后的结构必须仍然忠实于原文的所有内容，不遗漏、不添加
### 第三步：标题创作（核心环节）
 
标题是文章最重要的部分——决定了读者会不会点进来。必须在标题上花足够的心思。
 
#### 标题创作原则
 
1. **制造信息缺口**：标题要让读者觉得"不点进去就亏了"。给出一半信息，留一半悬念。避免在标题中把结论说完——说完了就没人点了
2. **要有逻辑性**：标题本身要是一个完整的、说得通的短句或短语，不能为了吸引眼球而语法混乱或拼凑关键词
3. **要有画面感或冲突感**：好的标题让人脑子里能浮现出一个场景，或者感受到一种张力。比如"我爸退休后干的第一件事，是让 AI 帮他炒股"就比"AI 与投资"有画面得多
4. **不要标题党**：吸引人和标题党的区别在于——点进去之后，内容是否兑现了标题的承诺。标题可以夸张，但不能欺骗
5. **三个平台的标题必须各不相同**：不是改几个字，而是从不同角度切入，适配各平台用户的阅读心理
#### 标题创作方法
 
每个平台至少构思 3 个备选标题，从中选出最好的一个。构思时可以从以下角度切入：
 
- **悬念型**：抛出一个让人好奇的问题或半句话（"录了一小时才发现，声音全废了"）
- **反差型**：利用身份、行为、结果之间的反差（"60 岁的老股民，开始写代码了"）
- **场景型**：用一个具体场景把人拉进去（"口袋里的手机录了一路，打开一听全是杂音"）
- **观点型**：直接亮出一个锐利的判断（"搜索引擎正在变成上一代的产品"）
- **共鸣型**：说出读者心里想说但没说出来的话（"AI 迟早会像空气一样，没人再提它"）
#### 各平台标题适配
 
- **微信公众号**：可以稍长（15-20 字），信息量大一些，适合叙事型或悬念型标题
- **X（推特）**：要短要锐（10-18 字），适合观点型或反差型，在信息流中一眼抓住人
- **小红书**：口语化、有温度（12-20 字），适合场景型或共鸣型，像朋友在跟你聊天
### 第四步：生成三个平台版本
 
#### 1. 微信公众号版
 
- **标题**：遵循第三步的标题创作方法论，从备选中挑出最佳标题。15-20 字，信息量大，有悬念或叙事感
- **风格**：根据文章内容灵活调整——
  - 严肃话题 → 正式但不生硬
  - 生活/科技话题 → 轻松、有温度
  - 观点类文章 → 有立场但不偏激
- **排版要求**：
  - 段落短一些（手机阅读体验），每段 3-5 句为宜
  - 适当使用小标题分隔章节
  - 开头要有吸引力（金句、提问、场景描写等）
  - 结尾可以设置互动引导（但不要生硬）
- **长度**：保持与原文相当，不需要大幅压缩
#### 2. X（推特）版
 
- **标题**：遵循第三步的标题创作方法论。10-18 字，短且锐利，适合信息流中一眼抓人
- **形式判断规则**：
  - 如果优化后的核心内容能在 **280 字符**（约 140 个中文字）以内表达清楚 → 生成**单条推文**
  - 如果内容较长或有多个要点 → 生成 **thread**（连续推文），每条推文标注序号如 1/5, 2/5...
- **风格**：
  - 简洁有力，每句话都要有信息量
  - 可以用更直接、更锐利的表达
  - 适合碎片化阅读
- **Thread 规则**（如适用）：
  - 第一条是 hook，要抓人
  - 每条推文独立成段，有完整意思
  - 最后一条可以总结或呼吁行动
#### 3. 小红书版
 
- **标题**：遵循第三步的标题创作方法论。12-20 字，口语化、有温度，像朋友在聊天
- **风格**：
  - 保持简洁，不使用 emoji 和话题标签
  - 口语化但不幼稚
  - 段落简短，适合手机滑动阅读
  - 用「」强调关键词
  - **⚠️ 禁止使用 `**文字**` 加粗标记**：小红书不支持 Markdown 语法，`**` 符号会原样显示为星号，破坏阅读体验。需要强调时只用「」，绝对不用 `**`
- **结构**：
  - 开头直接抛出核心观点或引发好奇
  - 中间用简短段落展开
  - 结尾一句话总结或引发思考
- **长度**：通常比公众号版短，控制在 300-800 字
### 第五步：生成配图 Prompt
 
为每个平台版本生成 **1 个配图的 AI 生图 prompt**，要求：
 
- **目标工具**：Midjourney（prompt 格式必须适配 Midjourney 语法）
- **语言**：英文
- **Midjourney 格式规范**：
  - 不要使用完整的英文句子，使用逗号分隔的短语/关键词风格
  - 在 prompt 末尾附加 Midjourney 参数，如 `--ar 16:9`（微信公众号封面）、`--ar 1:1`（X/推特）、`--ar 3:4`（小红书）
  - 可酌情添加 `--style raw`、`--s` (stylize)、`--c` (chaos) 等参数
  - 将最重要的画面描述放在 prompt 开头（Midjourney 对前置词权重更高）
- **与文章内容紧密相关**：配图要能反映文章的核心主题或情感
- **有创意**：不要太直白（比如文章讲 AI 就画个机器人），要有隐喻、意象或独特视角
- **风格指引**：包含具体的画风关键词（photography, illustration, watercolor, cyberpunk, minimalist 等，视文章内容而定）
- **构图和细节**：包含构图、色调、光影等关键词
- **格式**：每个 prompt 控制在 30-80 词（英文短语），加上末尾参数
三个平台的配图 prompt 应该各不相同，反映各平台的视觉风格差异：
- 微信公众号：可以更正式、有质感（如杂志封面风格）
- X：更抢眼、视觉冲击力强（信息流中要脱颖而出）
- 小红书：更生活化、有审美感（符合小红书的视觉调性）
### 第六步：生成微信公众号排版 HTML
 
完成前五步后，额外生成一个排版精美的 HTML 文件，让用户可以直接复制粘贴到微信公众号编辑器。
 
**操作方法**：读取 `references/wechat-format-guide.md`，按照其中的排版规范、HTML 模板和微信兼容性规则，将微信公众号版的标题 + 正文转换为全 inline style 的 HTML 文件。
 
**关键要求**：
- 只做排版，不改任何文字内容
- 所有 CSS 必须写成 inline style（微信不支持 class/id/style 标签）
- 作者名固定「{{YOUR_NAME}}」，公众号名固定「{{YOUR_ACCOUNT_NAME}}」，日期取当天
- HTML 文件保存到 `/mnt/user-data/outputs/[主题关键词]_公众号排版.html`
### 第七步：生成 X Article 排版 HTML
 
基于微信公众号版的完整内容（标题 + 正文），生成一个适合 X（推特）Article 长文功能的排版 HTML 文件。
 
**操作方法**：读取 `references/x-article-format-guide.md`，按照其中的排版规范和设计风格，将公众号版内容转换为 X Article 风格的 HTML 文件。
 
**关键要求**：
- 使用微信公众号版的完整内容，不做删减
- 视觉风格对齐 X 平台（深色文字、蓝色 #1d9bf0 强调色、简洁现代）
- 小标题使用蓝色下划线样式，引用块使用蓝色左边框
- 作者署名用 @{{YOUR_X_HANDLE}}，日期用英文格式
- HTML 文件保存到 `/mnt/user-data/outputs/[主题关键词]_X_Article.html`
### 第八步：生成小红书长文排版 HTML
 
基于微信公众号版的完整内容（标题 + 正文），生成一个适合小红书长文笔记的排版 HTML 文件。
 
**操作方法**：读取 `references/xiaohongshu-format-guide.md`，按照其中的排版规范和设计风格，将公众号版内容转换为小红书长文风格的 HTML 文件。
 
**关键要求**：
- 使用微信公众号版的完整内容，不做删减
- 模拟手机窄屏阅读体验（预览容器 420px）
- 小标题用「」书名号包裹，引用卡片用小红书红色 #ff2442 左边框
- 风格温暖、口语化，适合滑动阅读
- 不使用 emoji
- HTML 文件保存到 `/mnt/user-data/outputs/[主题关键词]_小红书长文.html`

## 输出格式
 
生成一个精简的 Markdown 文件，**只包含配图 Prompt 和标题备选**——正文内容已在三份 HTML 排版文件中，不要在 MD 里重复。结构如下：
 
```markdown
# 文章优化结果

## 优化摘要

> 简要说明做了哪些主要优化（2-3 句话）

---

## 标题备选

### 微信公众号（最终选用：[选用标题]）
1. [备选标题1]
2. [备选标题2]
3. [备选标题3]

### X（推特）（最终选用：[选用标题]）
1. [备选标题1]
2. [备选标题2]
3. [备选标题3]

### 小红书（最终选用：[选用标题]）
1. [备选标题1]
2. [备选标题2]
3. [备选标题3]

---

## 配图 Prompt

### 微信公众号（16:9）
> [英文 AI 生图 prompt --ar 16:9]

### X（推特）（1:1）
> [英文 AI 生图 prompt --ar 1:1]

### 小红书（3:4）
> [英文 AI 生图 prompt --ar 3:4]
```

## 输出流程
 
1. 先在对话中简要告诉用户做了哪些关键优化（1-2 句话即可）
2. 生成 Markdown 文件保存到 `/mnt/user-data/outputs/` 目录
3. 生成微信公众号排版 HTML 文件（参照 `references/wechat-format-guide.md`）
4. 生成 X Article 排版 HTML 文件（参照 `references/x-article-format-guide.md`）
5. 生成小红书长文排版 HTML 文件（参照 `references/xiaohongshu-format-guide.md`）
6. 使用 `present_files` 工具将所有文件提供给用户下载——公众号 HTML 排第一位，X Article HTML 第二，小红书长文 HTML 第三，Markdown 文件最后
7. 文件命名格式：
   - Markdown：`[主题关键词]_优化版.md`（如 `AI未来_优化版.md`）
   - 公众号 HTML：`[主题关键词]_公众号排版.html`（如 `AI未来_公众号排版.html`）
   - X Article HTML：`[主题关键词]_X_Article.html`（如 `AI未来_X_Article.html`）
   - 小红书长文 HTML：`[主题关键词]_小红书长文.html`（如 `AI未来_小红书长文.html`）
8. **标点自检**:在 `present_files` 之前,对四个文件各扫一眼,确认中文正文里没有残留的半角 `,` `。(句末)` `:` `;` `?` `!`。详见「中文标点规范」章节的自检清单
9. 简要告知用户：可以在 artifact 面板预览排版效果；如需粘贴到各平台，在浏览器中打开对应 HTML 文件后选中文章内容复制即可

## 注意事项
 
- **⚠️ 中文标点(最高优先级)**:所有中文语境下的标点必须使用全角中文标点,禁止出现半角英文标点。详见下文「中文标点规范」章节。这条规则适用于所有输出:Markdown 文件和三份 HTML 排版文件里的正文内容都必须遵守
- **英文词汇加中文注释**:文中出现的英文单词或短语(如 thread、hook、roadmap 等)需要在后面用括号附上简短的中文解释,例如"thread(连续推文)"。但英文产品名称、品牌名、专有名词(如 ChatGPT、Midjourney、OpenClaw、GitHub、Linux 等)不需要加注释,保持原样即可
- **不要过度改写**：用户是原作者，要尊重原文的个人风格和观点
- **不要添加原文没有的观点**：可以补充衔接语，但不要添加新论点
- **不要使用 AI 味重的表达**：避免"在当今社会"、"不言而喻"、"值得注意的是"等套话
- **标题是最高优先级**：标题决定了有没有人会看这篇文章。必须严格遵循第三步的标题创作方法论，每个平台至少构思 3 个备选，选出最有吸引力且逻辑通顺的一个。三个平台的标题必须从不同角度切入
- **配图 prompt 要具体**：不要写泛泛的描述，要有具体的画面感和创意方向

## 中文标点规范
 
这是一条**贯穿整个技能**的硬性规则。无论是生成 Markdown 文件、微信公众号 HTML、X Article HTML、还是小红书长文 HTML,中文正文内容里的标点必须全部使用全角中文标点,严禁出现半角英文标点。
 
### 必须使用中文标点的字符
 
| 场景 | 用这个(全角) | 不用这个(半角) | Unicode |
|------|---|---|---|
| 逗号 | ,| , | U+FF0C |
| 句号 | 。 | . | U+3002 |
| 冒号 | : | : | U+FF1A |
| 分号 | ; | ; | U+FF1B |
| 问号 | ? | ? | U+FF1F |
| 感叹号 | ! | ! | U+FF01 |
| 左括号(中文内容括注) | ( | ( | U+FF08 |
| 右括号(中文内容括注) | ) | ) | U+FF09 |
| 引号(直接引语) | "" | "" | — |
| 引号(书名/强调/特殊词) | 「」/《》 | "" | — |
| 破折号 | —— | -- | U+2014 ×2 |
| 省略号 | …… | ... | U+2026 ×2 |
| 顿号 | 、 | , | U+3001 |
 
### 中英文混排的具体规则
 
**1. 中文正文里的标点 → 全角**
- ✓ `昨天晚上,我们做了一场聊天。`
- ✗ `昨天晚上, 我们做了一场聊天.`
**2. 英文词内部或纯英文片段 → 保留半角**
- ✓ `AI、GitHub、Anthropic` (专有名词保留)
- ✓ 配图 prompt 的英文内容(`warm backlit scene, golden hour`)完全用半角
- ✓ HTML/CSS 代码(`style="font-size: 17px; color: #333;"`)完全用半角
- ✓ URL(`https://example.com/path?q=1`)完全用半角
**3. 中英混排的边界:以语境判断**
- ✓ `对 AI 又感兴趣` — 中英之间保留半角空格,这是中文排版最佳实践
- ✓ `SIP(系统完整性保护)` — 英文缩写后跟中文解释,用全角括号
- ✓ `Discord(软件名)` — 括号内是中文,用全角
- ✗ `WeChat(微信)` 或 `WeChat(WeChat)`:原则是"**括号内是中文就用全角括号**";括号内全英文则保持半角
- ✓ `近 1500 人` — 中文与数字之间保留半角空格
**4. Markdown/HTML 结构字符绝对不动**
- Markdown 的 `-` `#` `**` `>` `[]()` 全部保持半角原样
- HTML 标签、属性、CSS 值里的所有标点都保持半角原样
**5. 顿号与逗号的区别**
- 并列词:`思维导图、网球、摄影` — 用顿号
- 分句之间:`我喜欢思维导图,也喜欢摄影` — 用逗号
### 输出前的自检清单
 
生成 Markdown 或 HTML 前,对每一段中文正文做以下扫描:
 
1. 段落里任何"中文字 + `,`"的组合是不是已经变成"中文字 + `,`"?
2. 段落末尾的 `.` 是不是 `。`?
3. 冒号引出引语/列表/说明时,是不是 `:`?
4. 并列词之间用了顿号 `、` 而不是逗号吗?
5. 直接引语/强调词用的是 `""` 或 `「」`,不是 `""`?
6. 中文内容里的括注用的是 `()` 还是 `()`?
7. HTML 的 `style="..."` 属性里 CSS 的分号和冒号都还是**半角**的吗?(这些是 CSS 语法,不能换!)
### 工具辅助(可选)
 
如果不放心,可以用以下 Python 脚本在输出文件上跑一遍做校验/修正。规则:
 
- 只在"中文字符(或中文标点)紧邻的半角标点"处替换,避开英文/代码/URL
- 括号以"内部含中文字符"为判据:含中文→全角,不含→保留半角
- 水平空白`[ \t]`会被吞掉(正常),但换行符保留(不会破坏 Markdown 列表)
```python
import re, sys
 
CJK = r'[一-鿿㐀-䶿]'
CJK_PUNCT = (
    r'[，。；：？！'
    r'「」『』《》、'
    r'“”‘’（）…—]'
)
HS = r'[ \t]'
 
def convert(text):
    # 括号先处理(内部含中文→全角)
    text = re.sub(rf'\(([^()\n]{{1,120}})\)',
        lambda m: f'（{m.group(1)}）' if re.search(CJK, m.group(1)) else m.group(0),
        text)
    # 中文紧邻的标点换全角
    for half, full in [(',', '，'), (';', '；'), (':', '：'),
                        ('\\?', '？'), ('!', '！')]:
        text = re.sub(rf'({CJK}|{CJK_PUNCT}){HS}*{half}{HS}*',
                      lambda m, f=full: m.group(1) + f, text)
    # 句号(谨慎:只在中文+.+(空格/换行/中文/标签)时换)
    text = re.sub(rf'({CJK}){HS}*\.(?=[ \t<]|$|\n|{CJK})',
                  lambda m: m.group(1) + '。', text, flags=re.MULTILINE)
    # 修复 CSS 里 "); " 可能被吞掉的空格
    text = re.sub(r'\);([a-z])', r'); \1', text)
    return text
 
if __name__ == '__main__':
    for p in sys.argv[1:]:
        with open(p, encoding='utf-8') as f: s = f.read()
        with open(p, 'w', encoding='utf-8') as f: f.write(convert(s))
```
 
### 为什么这条规则重要
 
- 中文排版里半角标点(特别是逗号、句号)视觉上与文字贴得太紧,破坏阅读节奏
- 微信公众号、小红书、X 这些平台的阅读者大都是中文母语读者,半角标点会立刻显得"业余"或"机翻感"
- 混用半角和全角是**最糟糕的**——比全部半角还差,因为会让文章看起来没有经过校对
